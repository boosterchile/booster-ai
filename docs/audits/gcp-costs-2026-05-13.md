## 4. Recomendaciones priorizadas (con impacto $/mes)

> Cada recomendación incluye: acción concreta, archivo a tocar, impacto USD/mes, esfuerzo, riesgo.

### 4.0 P0 — Habilitar billing export a BigQuery (sin esto, todo lo demás es ciego)

- **Acción**: crear dataset `billing_export` en `booster-ai-494222` (o nuevo proyecto `boosterchile-billing`) y habilitar standard + detailed export del billing account.
- **Cómo**: vía consola (`Billing → Billing export`) o vía `google_billing_account_iam_member` + manual setup (el export en sí no es Terraform-able full, pero el dataset sí).
- **Esfuerzo**: 15 minutos.
- **Riesgo**: cero.
- **Impacto**: habilita **toda auditoría futura**. Datos empiezan a llegar dentro de ~24h.
- **Costo**: BigQuery storage del export ~$0.02/GB/mes; queries con free tier.
- **Próximo paso post-merge**: re-ejecutar esta auditoría con cifras reales por SKU + label.

### 4.1 P1 — Apagar el clúster GKE DR mientras no haya workload productivo

- **Decisión a tomar (PRESENTAR A FELIPE)**: ¿el DR cluster es requisito de auditoría/compliance (ISO/Corfo) **HOY**, o puede esperar hasta que telemetry-tcp-gateway esté en producción?
- **Opción A — Eliminar `infrastructure/dr-region.tf`**:
  - Ahorro: **~USD 100-150/mes (CLP 94k-141k)**
  - Esfuerzo: 1 PR, `terraform apply` ~10 min
  - Riesgo: si hay un evento DR mientras está apagado, RTO se va de minutos a horas (rebuild desde Terraform)
  - **Recomendado si no hay compromiso DR con cliente/Corfo** todavía
- **Opción B — Dejarlo creado pero con `node_count=0` / scale-to-zero forzado**:
  - Ahorro: parcial (~USD 50/mes), aún pagas cluster fee
  - Riesgo: bajo
- **Opción C — Mantener tal cual** (statu quo):
  - Costo: ~USD 100-150/mes
  - Justificación: solo si DR debe estar caliente

### 4.2 P1 — Downsize Cloud SQL a `db-custom-1-3840`

- **Acción**: bajar de `db-custom-2-7680` (2 vCPU / 7.68 GB) → `db-custom-1-3840` (1 vCPU / 3.84 GB), manteniendo REGIONAL HA + 50 GB SSD.
- **Justificación**: 3.9% CPU avg + 47.6% RAM avg. Tier nuevo deja headroom 25× CPU y 1.05× RAM.
- **Archivo**: `infrastructure/data.tf` (o donde viva `google_sql_database_instance.main`).
- **Esfuerzo**: 1 PR. `terraform apply` causa downtime ~3-5 min de la BD (lo programa Cloud SQL, hay maintenance window).
- **Impacto**: **~USD 120-130/mes (CLP 113k-122k)**.
- **Riesgo**: bajo si se valida primero con un load test del matching-engine + telemetry-processor a tráfico de pitch. Reversible en 5 min.
- **Mitigación**: hacerlo el mismo día que la maintenance window del cliente (sábado AM).

### 4.3 P1 — Downgrade Memorystore Redis STANDARD_HA → BASIC

- **Acción**: bajar tier Redis de `STANDARD_HA` (1 GB) a `BASIC` (1 GB) — no HA, single replica.
- **Justificación**: uso real 0.6% (~6 MB). El HA está pensado para failover < 1 min; el BASIC reinicia en ~2-5 min en caso de falla. Para conversation store + OIDC token cache + rate limiting (usos actuales), el BASIC es ampliamente suficiente.
- **Archivo**: `infrastructure/compute.tf` (resource `google_redis_instance.main`).
- **Esfuerzo**: 1 PR. **Atención**: cambiar tier requiere recrear la instancia (todos los datos se pierden). Como solo guarda caché efímera, no es problema, pero hay que coordinar deploy.
- **Impacto**: **~USD 40/mes (CLP 37k)**.
- **Riesgo**: medio — si en producción Redis cae, hay ~2-5 min de degradación (chat SSE puede perderse, rate limiting se reinicia). Aceptable en este TRL.
- **Alternativa más conservadora**: mantener STANDARD_HA pero a 1 GB sigue siendo el mínimo. No hay downsizing posible bajo 1 GB en Redis Memorystore.

### 4.4 P1 — Bajar `min_instances` de Cloud Run web / marketing / whatsapp-bot / telemetry-processor

- **Acción**: en `infrastructure/compute.tf`, cambiar `min_instances = var.environment == "prod" ? 1 : 0` a `0` para los 4 servicios sub-1 RPS.
- **Justificación**: tráfico real <0.002 RPS. Cold start de Cloud Run en saw1 es ~1-2 segundos para Node — aceptable para landing/web/whatsapp webhook (Twilio reintenta) / telemetry-processor (Pub/Sub reintenta).
- **Excepción**: `booster-ai-api` (3 RPS sostenidos + es el endpoint crítico de la PWA) — **mantener min=1**.
- **Archivo**: `infrastructure/compute.tf` líneas de cada `module "service_*"`.
- **Esfuerzo**: 1 PR pequeño.
- **Impacto**: **~USD 60-80/mes (CLP 56k-75k)**.
- **Riesgo**: bajo. Mitigación: warmup endpoint + alerting si p95 latency cold-start > 3s.

## 6. Cómo reproducir esta auditoría

Una vez habilitado el billing export, los siguientes queries dan el grueso del análisis:

```sql
-- Top SKUs últimos 30 días (USD)
SELECT
  service.description AS service,
  sku.description AS sku,
  SUM(cost) AS cost_usd,
  SUM(usage.amount) AS usage_amount,
  usage.unit AS usage_unit
FROM `booster-ai-494222.billing_export.gcp_billing_export_v1_019461_C73CDE_DCE377`
WHERE usage_start_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
GROUP BY service, sku, usage_unit
ORDER BY cost_usd DESC
LIMIT 30;

-- Costo por proyecto / label environment
SELECT
  project.id AS project_id,
  (SELECT value FROM UNNEST(labels) WHERE key = 'environment') AS env,
  SUM(cost) AS cost_usd
FROM `booster-ai-494222.billing_export.gcp_billing_export_v1_019461_C73CDE_DCE377`
WHERE usage_start_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
GROUP BY project_id, env
ORDER BY cost_usd DESC;

-- Forecast 90d
SELECT *
FROM ML.FORECAST(MODEL `booster-ai-494222.cost_forecast.monthly_cost_model`, 3);
-- (requiere crear el modelo antes con ARIMA_PLUS)
```
