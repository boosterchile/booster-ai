# Devils-advocate review — sec-001-empresa-onboarding-gate-hotfix

## Round 1 — 2026-05-29T00:00:00Z

Stance: assume the hotfix does NOT close the vector until each load-bearing claim survives. Weighed against "minimal correct closure," not "solves everything."

### Findings

#### P0-1 — The SC-4 CI harness, as specified (mirror of `check-is-demo-wire-completeness.ts`), is structurally blind to the very vector it claims to backstop.
The existing pattern (`apps/api/scripts/check-is-demo-wire-completeness.ts`) is a regex that only matches `app.use('/path', ...)` calls in `server.ts` and scans the arg block for `*Middleware` identifiers. The `/empresas/onboarding` hole was NOT "firebaseAuth present, userContext missing on an app.use" in a way a path-prefix scan would flag as anomalous — `/empresas/*` legitimately and intentionally omits userContext (the user does not exist yet). So the rule "firebaseAuth without userContext => must be allowlisted" would flag `/empresas/*` only as "needs an allowlist entry," and whoever shipped the hole would simply add the allowlist entry with rationale "user pre-onboarding" — exactly the true statement that makes the hole invisible. The harness converts a privilege-escalation gap into a one-line allowlist edit. It does not detect "this open route performs a privileged WRITE (creates an active dueño membership)."
Concrete false-negative classes the regex cannot see:
  - Middleware applied **inside the sub-router** (e.g. `meRouter.route(...)`, `assignmentsRouter.route('/', chatRouter)`), not via `app.use` in server.ts. The chat router and cobra-hoy routers inherit auth from parent mounts; a privileged write added inside one of those sub-routers is invisible to a server.ts `app.use` scan.
  - The in-handler gate this very hotfix adds. The 403 lives in `routes/empresas.ts`, NOT in `server.ts`. The harness parses only `server.ts`. So the harness cannot verify the gate it is supposed to protect even exists, and a future PR deleting the in-handler `if (!EMPRESA_SELF_ONBOARDING_ENABLED)` line keeps the build green.
  - Conditional/dynamic mounts (`if (opts.firebaseAuth) { ... }`, `if (config.INTERNAL_CRON_CALLER_SA)`) — the regex flattens these; a route mounted only under a condition still appears unconditionally.
  - The wrapper-middleware idiom (`/certificates/*` uses `async (c, next) => firebaseAuthMiddleware(c, next)`), which mentions the identifier inside a closure — coincidentally caught here, but a renamed/aliased middleware would not be.
**Remedy**: Do not sell SC-4 as "the durable backstop that replaces the never-deployed creation-time gate" (spec §3 SC-4, verbatim). That framing is the "documented but not verified" anti-pattern the spec itself condemns in Alternative B. Either (a) downgrade SC-4 to an honest smoke check with an explicit "ACCEPTABLE DEFEAT SCENARIOS" block, exactly as `check-handler-completeness.ts` already does, or (b) make the durable backstop a **service-layer** assertion (see P0-2) plus a behavioural test that the gate returns 403 when the flag is false — a behavioural test is a real backstop; a static server.ts scan is not. The harness as written gives false confidence.

#### P0-2 — Gate placement in the route handler leaves `onboardEmpresa` itself unguarded; defense-in-depth is absent and the spec's own C-1 forbids fixing it.
`onboardEmpresa` (`services/onboarding.ts:67`) is the function that actually writes `users.status='activo'` + `empresas` + `memberships role='dueno' status='activa'`. The flag check sits in `routes/empresas.ts`, before the call. Confirmed: today `onboardEmpresa` has exactly one non-test caller (`routes/empresas.ts:38`). So the gate closes the current vector. BUT the spec elevates a route-handler check to "the" closure while SC-3's own audit doc and the follow-up redesign explicitly anticipate a NEW approved→dueño path being built. The moment that path lands (the redesign will call `onboardEmpresa` or a sibling from `approveSignupRequest`), the route-handler flag does nothing for it, and there is no invariant in the service preventing an ungated caller. The cheapest durable closure is a guard *in the service*: `onboardEmpresa` should refuse (throw) when `EMPRESA_SELF_ONBOARDING_ENABLED` is false, OR — better — take an explicit `reason: 'self_service' | 'admin_approved'` discriminator so the flag only blocks self-service and the future approval path is structurally distinct. C-1 ("no touching onboardEmpresa's logic") actively prevents the more defensible fix and should be challenged, not accepted.
**Remedy**: Add the flag check (or a required `provisioningContext` arg) inside `onboardEmpresa` in addition to the handler. Relax C-1 to "no change to the DB write semantics" rather than "no touching the function." Cost: ~5 lines + one test. Still minimal, and it is the difference between "door closed today" and "door cannot be reopened by accident."

#### P1-3 — The flag-wiring mechanism is under-specified and the spec references an identifier (`appConfig`) that the target file does not import.
`routes/empresas.ts` imports nothing from config. `createEmpresaRoutes({ db, logger })` has no config in its opts. The repo's established pattern is a direct module import `import { config as appConfig } from '../config.js'` (as in `admin-signup-requests.ts:8`, `admin-seed.ts:5`). The spec's §7.2 / SC-2 writes `!appConfig.EMPRESA_SELF_ONBOARDING_ENABLED` as if `appConfig` is already in scope — it is not. Fine to resolve at /plan, but note: a direct module import couples the route to the singleton `config`, which makes the SC-5(a)/(b) tests (flag on vs off) harder — you cannot flip the flag per-test without module mocking or `vi.stubEnv` + re-import. `admin-signup-requests` has this same testability friction.
**Remedy**: Decide explicitly at /plan: pass the flag (or `{ selfOnboardingEnabled: boolean }`) through `createEmpresaRoutes` opts (injectable, trivially testable, matches the dependency-injection style of `createServer`), rather than a bare module import. Update SC-2 wording so the spec does not reference a non-existent binding.

#### P1-4 — SC-3 route-audit claims `/me/*` is "INTENTIONAL-OPEN, self-scoped" but the audit must verify each `/me` sub-route, not the group.
`server.ts:290-317` mounts `/me` and `/me/*` on firebaseAuth WITHOUT userContext, then selectively adds userContext only on `/me/push-subscription`, `/me/cobra-hoy/*`, `/me/liquidaciones`. The sub-routers `createMeClaveNumericaRoutes`, `createMeConsentsRoutes` run with firebaseAuth only and resolve userId from firebase_uid inside the handler. `me-consents.ts:106` checks `role==='dueno'||'admin'` — i.e. a `/me/*` route reads membership/role data. The SC-3 definition of INTENTIONAL-OPEN ("serves no cross-tenant data and grants no privilege absent an in-handler allowlist/approval check") is a claim that must be verified per sub-route, not asserted for the `/me` group. The clave-numerica route sets/rotates a numeric login credential for any logged-in user including pre-onboarding (server.ts:306-308) — confirm it cannot set another user's clave (IDOR) before stamping `/me/*` as INTENTIONAL-OPEN.
**Remedy**: SC-3 must enumerate `/me/clave-numerica`, `/me/consents`, `/me/push-subscription`, `/me/cobra-hoy/*`, `/me/liquidaciones` as **separate rows** with per-route verdicts, not one `/me/* = INTENTIONAL-OPEN` row. Otherwise the audit launders an unreviewed sub-route through a group label — the same mistake that hid the onboarding hole.

#### P1-5 — "Forensics found no exploitation" is load-bearing for the no-incident posture but the evidence is deferred to a doc that does not exist yet, and one cited metric is a category error.
Spec §2 and §13 assert "0 solicitudes_registro; all 7 dueños predate the approval flow and are PO/pilot/demo." The forensic doc (`evidence/forensic-blast-radius.md`) is "to be written with PR." The claim that the 7 dueños are all benign rests on identity attribution. Google `signInWithPopup` is live and the hole is open; an attacker who self-onboarded would ALSO appear as a `memberships role='dueno'` row with a Firebase uid — there is no `solicitudes_registro` row for the onboarding path because onboarding bypasses signup-requests entirely. So "0 solicitudes_registro" is NOT evidence that no self-onboarding occurred — those are different tables. The real evidence is: enumerate all `usuarios` + `membresias role='dueno'` rows, cross-check each firebase_uid against known PO/pilot/demo identities AND against the Firebase Auth user list (creation timestamp, provider=google.com, email domain).
**Remedy**: Make the forensic query explicit and run it BEFORE ship, not "with PR": `SELECT u.email, u.firebase_uid, u.creado_en, m.rol FROM usuarios u JOIN membresias m ON ... WHERE m.rol='dueno'` joined against Firebase Auth provider+createdAt. Label the evidence as DB-output, not vibes. The "0 solicitudes_registro" sentence is a category error and should be struck or corrected — it measures the wrong table.

#### P2-6 — Blast radius of flag-default-false on the web client is asserted ("existing dueños unaffected") but not verified against the web onboarding call site.
`apps/web/src/hooks/use-onboarding-mutation.ts` exists (grep hit). The spec asserts the web app only hits `/empresas/onboarding` for new-owner signup, but does not cite the call site or confirm no existing-user code path (e.g. a "complete your profile" re-entry, a retry-after-error, or a multi-empresa "add another company" flow) invokes it. If a logged-in user without a `usuarios` row (post-Firebase-signup, pre-onboarding — a state the codebase explicitly supports per the `/me` pre-onboarding comments) hits the now-403'd endpoint, they are bricked: authenticated in Firebase, no DB row, no path forward, and the redesign that gives them a path is deferred. The R-4 mitigation ("manual provisioning") assumes Booster knows who they are; a self-signed-up Google user stuck at 403 is invisible.
**Remedy**: SC-2's 403 body must include actionable next-step copy (it does — "directing them to request access"), AND the web client must be checked: does the onboarding form render for users who can never complete it now? Add to scope a one-line web guard or a confirmation that the web flow already routes such users to the signup-request/contact path. Also confirm no e2e/Playwright spec exercises `/empresas/onboarding` happy-path against staging where the flag will be false (it will fail). Grep `test:e2e` specs for onboarding before ship.

#### P2-7 — Rollback semantics are self-contradicting by design and should be stated as "this flag has no safe on-state in prod."
§11 says "Rollback: set flag true to restore prior (open) behaviour" then immediately "Note: reopening restores the vuln." So the flag's `true` state is never safe to use in prod until the redesign ships. That means this is not really a reversible feature flag — it is a kill switch with a permanently-disabled "on." Calling it a feature flag with a rollout plan (§11 "Reopen by setting true") overstates reversibility.
**Remedy**: Rename the mental model in the spec: this is a **kill switch**, on-state forbidden in prod until the redesign lands a gated path. The only legitimate prod rollback for a bug in the gate is revert-the-PR (spec already says this — good). State plainly: "EMPRESA_SELF_ONBOARDING_ENABLED=true in prod is prohibited by policy until onboarding-flow-redesign ships." Otherwise a future operator reads "rollback = flag true" and reopens the hole during an unrelated incident.

### Seven-axis coverage
- **Premise**: "Onboarding route is the sole unprivileged dueño-creation path" — VERIFIED (only non-test caller of onboardEmpresa; seedDemo/admin-seed are platform-admin + DEMO_MODE gated; demo-login reads not writes; admin approve uses Admin SDK createUser but is platform-admin gated). Premise holds. The dangerous implicit premise is P0-2: that a route-handler check is a durable closure.
- **Scope/second-order**: P2-6 (web client + e2e), P1-4 (/me sub-routes).
- **Alternatives**: A/B/C considered and documented — adequate. Not considered: service-layer guard (P0-2), kill-switch framing (P2-7).
- **Failure modes**: F1 stuck-pre-onboarding user (P2-6); F2 future approval path calls onboardEmpresa unguarded (P0-2); F3 harness false-negative ships next hole (P0-1).
- **Reversibility**: P2-7 — on-state unsafe; revert-PR is the real rollback.
- **Drift signals**: heavy deferral to follow-up — deferrals are justified and ledgered in §5/§13. No unjustified drift introduced by the hotfix scope. Acceptable.
- **Evidence quality**: P1-5 — "0 solicitudes_registro" is a category error; forensic query deferred and measures the wrong table. Weakest evidence axis.

### Verdict: APPROVE_WITH_RESERVATIONS

The hotfix DOES close the live unprivileged vector today — the gate is correctly placed before any DB write and `/empresas/onboarding` is genuinely the only unprivileged path to dueno creation. That is the minimal correct closure and it is real.

Reservations that must be addressed before ship (not after):
- **P0-1**: Stop framing SC-4 as the durable backstop. Either an honest smoke check OR a behavioural test of the gate. The static server.ts scan does not detect privileged writes and would re-admit this very hole via an allowlist entry.
- **P0-2**: Add the flag/context guard inside `onboardEmpresa` (relax C-1). Without it, the closure is route-deep, not vector-deep, and the in-flight redesign reopens it.
- **P1-5**: Run the actual forensic DB query before ship; correct the "0 solicitudes_registro" category error.

Residual risks to accept and document: P2-6 stuck-user UX (mitigated by 403 copy + manual provisioning, but verify the web call site), P2-7 kill-switch framing.

Out of scope for this review: the redesign itself, the 409 approve↔onboarding conflict, demo/app prospect strategy — correctly deferred.
