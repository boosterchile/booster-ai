## Opciones para T1.3 (decisión PO)

### Opción A — Delete completo del archivo `cargo-request.ts`

**Implicaciones**:
- LOC delta: `-150` a `-200` (delete schema + remove barrel re-export + decidir si eliminar también `cargoRequestIdSchema`).
- Requiere update de 4 ADRs y 1 skill para reflejar que el concepto ya no existe en el código.
- Requiere update del comentario roadmap en `trip-request.ts`.
- Si PO decide implementar el flujo WhatsApp NLU futuro, hay que **re-crear** el schema (regreso del trabajo eliminado).
- **No recomendado** salvo decisión PO de abandonar el concepto.

### Opción B — Eliminar solo `cargoRequestStatusSchema` (el status enum)

**Implicaciones**:
- LOC delta: `-15` (solo el enum).
- Pero el resto del `cargoRequestSchema` (que tiene `status: cargoRequestStatusSchema`) queda roto.
- Requiere también reformar `cargoRequestSchema` para no depender del status — el concepto pierde coherencia.
- **No recomendado** — solución a medias.

### Opción C — Mantener todo + anotar explícitamente como TS-future (recomendado)

**Implicaciones**:
- Edit `domain/cargo-request.ts`: agregar docstring al inicio del archivo declarando explícitamente:
  - El concepto está documentado en ADR-006, ADR-008, ADR-010, skill empty-leg-matching.
  - La tabla SQL `cargo_requests` se materializará en slice 2 del flujo WhatsApp NLU (post-S1).
  - El `cargoRequestIdSchema` ya es FK en `trip.cargo_request_id` (concepto integrado, no orphan).
  - Caso 1 del inventory.md S1a queda como **TS-future legítimo**, no requiere refactor.
- LOC delta: `+20` (docstring explícito).
- Update `inventory-classification.md`: reclasificar Caso 1 de "TS-only-orphan" a "TS-future pending-materialization".
- Update `plan-s1a.md` T1.3: cambiar de "eliminar orphan" a "anotar TS-future + actualizar classification".

**Recomendación**: Opción C.
